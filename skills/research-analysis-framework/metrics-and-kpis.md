# Metrics & KPIs - Defining Success Indicators

## Purpose

Define measurable, quantifiable indicators that track whether a project/initiative is succeeding. Metrics are the language that turns qualitative assessments into objective measurements.

---

## Types of Metrics

### 1. INPUT METRICS (What We're Putting In)

**Definition**: Resources required to execute

| Metric | How to Measure | Example Target |
|--------|---|---|
| **Budget Required** | Total project cost | R$ 550,000 |
| **Personnel Required** | Full-time equivalents (FTE) | 5 FTE for 12 months |
| **Time Required** | Person-hours | 8,000 hours total |
| **Special Skills** | Count of specialized skills | 2 (structural + electrical engineer) |
| **Equipment/Tools** | Capital equipment needed | 3x excavators, 1x concrete truck |
| **Infrastructure** | Systems/systems needed | Rental yard, tool storage, office |

**Why Track**: 
- Ensures resources available before starting
- Prevents "oh, we also need X" mid-project
- Baseline for efficiency comparison

### 2. PROCESS METRICS (How We're Doing It)

**Definition**: How execution is progressing

| Metric | How to Measure | Example Target |
|--------|---|---|
| **Schedule Variance** | Planned timeline vs. actual (%) | Within 5% of timeline |
| **Budget Variance** | Planned cost vs. actual (%) | Within 10% of budget |
| **Milestone Completion** | On-time delivery of phases | 100% on-schedule |
| **Quality Incidents** | Count of rework, defects | Zero critical, <5 minor per month |
| **Cost per Unit** | Cost efficiency (e.g., per sq meter) | R$ 3800/m² (vs. benchmark R$ 4000) |
| **Productivity** | Output per person-hour | 2.5 sq meters per person-day |
| **Safety Incidents** | Count of injuries/near-misses | Zero lost-time incidents |

**Why Track**:
- Early warning if project drifting
- Cost control in real-time
- Quality assurance during execution
- Efficiency baseline for future projects

### 3. OUTPUT METRICS (What We're Producing)

**Definition**: Quality and specification of deliverable

| Metric | How to Measure | Example Target |
|--------|---|---|
| **Specification Compliance** | % of specs met | 100% compliance |
| **Quality Score** | Inspection/certification pass | Zero major defects |
| **Performance Targets** | Functional specifications | Roof load capacity 150 kg/m² |
| **Aesthetic Quality** | Visual inspection | Professional finish |
| **Completion Percentage** | % of work finished | 100% (no punch list items) |
| **Defect Density** | Defects per unit delivered | <2 per 1000 sq meters |
| **User Satisfaction** | Client approval rating | 9.5/10 or higher |

**Why Track**:
- Ensures delivery meets requirements
- Prevents low-quality project acceptance
- Client satisfaction indicator
- Basis for payment milestones

### 4. OUTCOME METRICS (Results after delivery)

**Definition**: Real-world value created

| Metric | How to Measure | Example Target |
|--------|---|---|
| **ROI** | Return on investment (%) | 15% annual return |
| **Payback Period** | Months to break even | 7 years (for residential) |
| **Cost Avoidance** | Savings vs. alternative | R$ 50,000 vs. renting |
| **Productivity Gain** | Improvement in output | 25% faster turnaround |
| **Market Impact** | Sales, market share | +20% customer base |
| **Asset Value** | If tangible asset created | House worth R$ 750k (vs. R$ 550k invested) |
| **User Satisfaction** | Client/customer feedback | 95% would recommend |

**Why Track**:
- Did project achieve intended benefit?
- Learning for future projects
- Justification for investment
- Basis for "lessons learned"

### 5. HEALTH METRICS (Overall project status)

**Definition**: Real-time project health signals

| Metric | How to Measure | Status Indicators |
|--------|---|---|
| **Schedule Health** | Days ahead/behind plan | 🟢 ahead, 🟡 on-track, 🔴 behind |
| **Budget Health** | Spent vs. planned | 🟢 under, 🟡 on-track, 🔴 over |
| **Quality Health** | Defect rate vs. plan | 🟢 low, 🟡 acceptable, 🔴 high |
| **Risk Status** | Active risks + mitigations | 🟢 controlled, 🟡 watched, 🔴 critical |
| **Stakeholder Health** | Satisfaction/confidence | 🟢 high, 🟡 stable, 🔴 concerned |

**Why Track**:
- One-glance project status
- Early warning system
- Decision point trigger ("time to escalate?")

## Defining a Good KPI

Use the **SMART Framework**:

```
S - SPECIFIC: Clearly defined, no ambiguity
E - MEASURABLE: Quantifiable, objective
A - ACHIEVABLE: Realistic target, based on evidence
R - RELEVANT: Connected to actual success
T - TIME-BOUND: Has deadline or measurement frequency
```

### Example: Good vs. Bad KPI

```
❌ BAD: "House construction will be high quality"
├─ Not specific (what is "quality"?)
├─ Not measurable (how do you measure this?)
└─ Not time-bound (when do we assess?)

✅ GOOD: "Inspection will certify 100% code compliance by Q1 2024"
├─ Specific (code compliance is defined)
├─ Measurable (inspector certifies yes/no)
├─ Time-bound (Q1 2024)
├─ Achievable (standard for new construction)
└─ Relevant (compliance is required for occupancy)
```

## By Project Stage

### PLANNING STAGE

| Metric | Purpose | Target |
|--------|---------|--------|
| Scope clarity | % of requirements documented | 100% by kickoff |
| Risk identification | # of risks identified | 15-20 documented |
| Resource planning | Budget approved vs. estimated | ±5% variance |
| Timeline approval | Schedule agreed with stakeholders | Signed-off |
| Assumption validation | # of critical assumptions tested | 100% before start |

### EXECUTION STAGE

| Metric | Purpose | Target |
|--------|---------|--------|
| Schedule variance | Days ahead/behind | Within ±3 days |
| Budget variance | Spent vs. planned | Within ±10% |
| Milestone completion | On-time delivery | 100% delivered |
| Quality incidents | Rework, defects | <5 per phase |
| Risk tracking | Active risks | All mitigated or escalated |
| Stakeholder engagement | Status reporting | Weekly communication |

### COMPLETION STAGE

| Metric | Purpose | Target |
|--------|---------|--------|
| Spec compliance | % of requirements delivered | 100% |
| Quality sign-off | Client acceptance | Formal approval |
| Punch list | Outstanding items | Zero or negligible |
| Final budget | Total cost vs. planned | Within ±5% |
| Timeline | Actual vs. planned completion | Within ±1 week |
| Lessons learned | Process improvements | Documented for next project |

## Building Your Metrics Dashboard

### Step 1: Identify Success Dimensions

For your project:
```
What does "success" look like?
├─ On-time? (Schedule)
├─ On-budget? (Cost)
├─ High-quality? (Quality)
├─ Safe? (Safety)
├─ Profitable? (Financial)
└─ Satisfying? (Stakeholder)
```

### Step 2: Define 1-2 Metrics per Dimension

```
Schedule Success:
├─ KPI: Milestone completion on-time
└─ Target: 100% delivered within ±3 days

Budget Success:
├─ KPI: Cost variance
└─ Target: ±10% of planned budget
```

### Step 3: Set Targets Based on Evidence

```
Don't guess. Use:
├─ Industry benchmarks
├─ Historical projects
├─ Market data
├─ Expert estimates
└─ Comparable projects
```

### Step 4: Establish Measurement Frequency

```
Schedule: Weekly (real-time progress)
Budget: Bi-weekly (cost trending)
Quality: Daily (inspection findings)
Stakeholder: Monthly (status report)
Safety: Daily (incident log)
```

### Step 5: Create Dashboard

```
PROJECT STATUS DASHBOARD - Week 12

Schedule:      🟡 On-track (2 days ahead)
Budget:        🟢 Under budget (8% savings)
Quality:       🟡 3 minor defects (rework planned)
Safety:        🟢 Zero incidents
Risks:         🟡 1 active (mitigation in place)
Stakeholder:   🟢 Satisfied (latest survey 8.5/10)

Trend: Green all week. Minor quality items being addressed.
Status: On track for delivery Q1 2024.
```

## Common Metrics by Project Type

### CONSTRUCTION PROJECT

```
Schedule:      % of milestones on-time
Budget:        Cost variance ($ vs. plan)
Quality:       Code compliance score (%)
Safety:        Lost-time incidents (count)
Stakeholder:   Client satisfaction (1-10)
Performance:   Building certification pass/fail
```

### SOFTWARE PROJECT

```
Schedule:      Sprints completed on-time (%)
Budget:        Development cost per feature
Quality:       Defect density (per 1000 LOC)
Performance:   System uptime (%)
Stakeholder:   Feature adoption (%)
Value:         Support ticket reduction (%)
```

### BUSINESS INITIATIVE

```
Schedule:      Initiative milestones on-time (%)
Budget:        ROI achieved (%)
Quality:       Process compliance (%)
Adoption:      User adoption rate (%)
Impact:        Revenue/cost impact ($)
Stakeholder:   Executive satisfaction (1-10)
```

## Quality Checklist

For each KPI you define:

- [ ] Written as SMART (Specific, Measurable, Achievable, Relevant, Time-bound)
- [ ] Baseline established (where are we today?)
- [ ] Target is realistic (based on evidence/benchmarks?)
- [ ] Owner assigned (who tracks this?)
- [ ] Frequency clear (how often measured?)
- [ ] Data source identified (where does this come from?)
- [ ] Threshold set (what's acceptable, warning, critical?)
- [ ] Tied to decision (what action if threshold breached?)
- [ ] Reviewed regularly (monthly/quarterly check-in)
- [ ] Connected to overall success (not vanity metric)
